# Milestone v1.0: Self-Hosted Operator Product

**Status:** ✅ SHIPPED 2026-03-17
**Phases:** 1-5
**Total Plans:** 15

## Overview

Nebula v1.0 turned an early gateway prototype into a self-hosted operator product with a credible deployment path, a dedicated admin console, routed-playground and observability workflows, hardened governance behavior, and benchmark-backed proof materials for demos and academic evaluation.

## Phases

### Phase 1: Self-Hosted Foundation

**Goal**: Nebula can be deployed and operated through a credible self-hosted path with explicit runtime health and durable persistence direction.  
**Depends on**: Nothing (first phase)  
**Plans**: 3 plans

Plans:

- [x] 01-01: Define the self-hosted packaging and runtime topology
- [x] 01-02: Harden configuration, secrets, and health/readiness behavior
- [x] 01-03: Introduce a durable persistence and migration path

**Details:**
- Canonical Docker image and Docker Compose topology for Nebula, PostgreSQL, and Qdrant
- Non-local runtime guardrails, readiness endpoints, and degraded dependency reporting
- Alembic-backed governance persistence with PostgreSQL-first self-hosted startup

### Phase 2: Operator Console

**Goal**: Operators can manage the deployment through a focused web console instead of raw API calls only.  
**Depends on**: Phase 1  
**Plans**: 3 plans

Plans:

- [x] 02-01: Stand up the frontend shell and admin authentication model
- [x] 02-02: Implement tenant and API key workflows
- [x] 02-03: Implement tenant policy management workflows

**Details:**
- Dedicated Next.js operator console with memory-only admin session behavior
- Tenant CRUD and API-key issuance/revocation workflows
- Structured policy editing backed by backend-authored metadata

### Phase 3: Playground & Observability

**Goal**: Nebula visibly demonstrates how it routes, caches, falls back, and records request outcomes.  
**Depends on**: Phase 2  
**Plans**: 3 plans

## Milestone Summary

**Key Accomplishments**

- Shipped a self-hosted deployment story with explicit runtime safety, readiness checks, and migration-backed persistence.
- Added a dedicated operator console for authentication, tenants, API keys, and policy management.
- Shipped Playground and Observability workflows that expose real routing, fallback, ledger, and dependency-health behavior.
- Hardened governance behavior so denied and fallback-blocked requests remain explainable from both response metadata and ledger records.
- Produced benchmark-backed product proof, focused architecture/evaluation docs, and a repeatable pilot/demo package.

**Key Decisions**

- Docker Compose remained the single supported self-hosted entrypoint for v1.0.
- The operator console stayed same-origin and memory-authenticated instead of expanding browser persistence or backend CORS scope.
- Playground response metadata and persisted ledger outcome remained visibly separate to preserve operator clarity.
- Product proof stayed benchmark-led rather than adding a demo-only product narrative.

**Issues Deferred**

- None recorded at milestone closeout.

**Technical Debt Incurred**

- No milestone-blocking technical debt remained at archival time.
