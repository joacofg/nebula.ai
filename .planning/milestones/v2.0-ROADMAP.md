# Milestone v2.0: Hosted Control Plane Validation

**Status:** ✅ SHIPPED 2026-03-23
**Phases:** 6-10
**Total Plans:** 14

## Overview

Nebula v2.0 validated an optional hosted control plane layered onto the shipped self-hosted gateway. The milestone stayed narrow: hosted surfaces manage metadata and control intent, all linking remains outbound-only, local runtime enforcement stays authoritative, and the only remote-management capability is one audited low-blast-radius action that improves pilot credibility without weakening the trust boundary.

## Phases

### Phase 6: Trust Boundary and Hosted Contract

**Goal**: Operators can understand and trust the hybrid hosted/self-hosted boundary before any deployment is linked.  
**Depends on**: Phase 5  
**Plans**: 3 plans

Plans:

- [x] 06-01: Freeze the hosted/local contract and exported metadata schema
- [x] 06-02: Define trust-boundary language, failure semantics, and disclosure surfaces
- [x] 06-03: Publish the hosted UI and architecture narrative for the boundary

**Details:**
- Hosted UI and architecture surfaces clearly distinguish hosted metadata and control intent from locally authoritative runtime-enforced state.
- Operators can inspect a concrete description of what data leaves the self-hosted environment by default.
- Hosted surfaces explicitly exclude raw prompts, responses, provider credentials, and authoritative runtime policy state from the default exported set.

### Phase 7: Deployment Enrollment and Identity

**Goal**: Self-hosted deployments can register with the hosted plane through an explicit outbound-only identity flow that operators can safely manage.  
**Depends on**: Phase 6  
**Plans**: 3 plans

Plans:

- [x] 07-01: Hosted registration: DB schema, enrollment service, admin API for slot creation and token generation
- [x] 07-02: Outbound linking: enrollment exchange endpoint, gateway startup hook, local identity storage
- [x] 07-03: Lifecycle management: revoke, unlink, relink endpoints and console deployment UI

**Details:**
- Operators can register a self-hosted Nebula deployment with the hosted control plane without manual database edits.
- A self-hosted gateway completes an outbound-only linking flow using short-lived enrollment credentials and deployment-scoped steady-state credentials.
- Operators can revoke, unlink, and relink a deployment without duplicate or ghost active inventory records.

### Phase 8: Fleet Inventory and Freshness Visibility

**Goal**: Operators can use the hosted plane for honest fleet visibility without confusing hosted freshness with local runtime authority.  
**Depends on**: Phase 7  
**Plans**: 3 plans

Plans:

- [x] 08-01: Hosted heartbeat ingest: models, DB migration, ingest service, freshness calculation, endpoint, and TypeScript types
- [x] 08-02: Gateway heartbeat sender: credential storage fix, background task, lifespan wiring, backend tests
- [x] 08-03: Frontend fleet inventory UI: FreshnessBadge, DependencyHealthPills, extended table and drawer

**Details:**
- Hosted inventory lists each linked deployment with identity, environment label, Nebula version, registration time, and last-seen time.
- Hosted views show connected, degraded, stale, or offline freshness semantics with explicit reason codes instead of overstated runtime-health claims.
- Hosted deployment detail surfaces capability and compatibility state and fails unsupported actions closed.

### Phase 9: Audited Remote Management

**Goal**: The hosted plane proves limited management value through one safe remote action that is pulled outbound and enforced locally.  
**Depends on**: Phase 8  
**Plans**: 3 plans

Plans:

- [x] 09-01: Define `rotate_deployment_credential` queue models, persistence, and hosted admin queue/history APIs
- [x] 09-02: Implement outbound polling, local authorization, and credential-rotation completion flow
- [x] 09-03: Add expiry-aware audit rollups and the hosted deployment-detail action/history UI

**Details:**
- Operators can queue one allowlisted, non-serving-impacting remote-management action for a linked deployment.
- A linked deployment polls for remote actions outbound, applies them only after local authorization checks, and records queued, applied, or failed outcomes.
- Remote-management actions expire safely, remain idempotent, and cannot directly mutate local runtime policy or provider credentials from the hosted plane.

### Phase 10: Pilot Proof and Failure-Safe Operations

**Goal**: Nebula can demonstrate that hosted-control-plane outages degrade visibility only and that the hybrid model is credible in pilot conversations.  
**Depends on**: Phase 9  
**Plans**: 2 plans

Plans:

- [x] 10-01: Prove hosted outages degrade visibility only while serving and readiness stay available
- [x] 10-02: Publish the pilot-ready trust-boundary, onboarding, and outage-safe narrative

**Details:**
- Loss of hosted control-plane connectivity does not break the self-hosted gateway's serving path and is surfaced as degraded or stale hosted visibility instead of runtime failure.
- Docs and demo materials explain hybrid onboarding, trust boundaries, remote-management safety limits, and hosted-control-plane outage behavior in a pilot-ready narrative.
- Pilot materials make explicit that the hosted plane is metadata-and-intent only while local enforcement remains authoritative.

## Milestone Summary

**Key Accomplishments**

- Locked the hosted trust boundary with a canonical metadata-export contract, public disclosure page, and aligned docs.
- Added outbound-only hosted deployment enrollment, durable identity, and lifecycle controls with a dedicated deployments console surface.
- Shipped honest fleet inventory with freshness semantics, dependency visibility, and fail-closed compatibility handling.
- Shipped one audited hosted-link credential rotation workflow with queue/history, local authorization, and live end-to-end validation.
- Proved hosted outages degrade visibility only and aligned the pilot narrative across UI, docs, and demo materials.

**Key Decisions**

- Hosted control plane remains metadata-and-intent only while local runtime enforcement stays authoritative.
- Linking remains outbound-only with short-lived enrollment bootstrap and deployment-scoped steady-state credentials.
- Remote management is limited to one audited, non-serving-impacting allowlisted action in v2.0.
- Public hosted-trust messaging stays schema-backed so UI and docs cannot drift.

**Issues Deferred**

- Future hosted operations remain deferred to a later milestone: approval-gated remote actions, richer diagnostics/remediation, and broader fleet orchestration.
- Hosted account-model expansion remains deferred: richer hosted user/org/RBAC workflows and public self-serve signup/billing.

**Technical Debt Incurred**

- No milestone-blocking technical debt remained at archival time.
