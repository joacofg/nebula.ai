# Knowledge

- S01 / API contract: Treat `docs/adoption-api-contract.md` as the only canonical public compatibility boundary. Update README and architecture docs by linking back to it rather than restating request/response details in multiple places.
- S01 / Validation behavior: `ChatCompletionRequest` now enforces the "at least one user message" invariant at schema-validation time, so invalid requests return FastAPI's native 422 `detail` payload and do not emit route-level `X-Nebula-*` headers.
- S01 / Test alignment: The admin Playground is an admin-only, non-streaming operator surface that can route differently from the public adoption path for the same requested model. Do not assume Playground responses should mirror public-route provider selection.
- S01 / Header assertions: For routed denials and fallback-blocked paths, assert that `X-Nebula-Policy-Outcome` is present and meaningful rather than hard-coding a specific string unless the runtime contract explicitly guarantees that exact value.
- S02 / Adoption docs composition: Keep `docs/quickstart.md` as the single happy-path self-hosted walkthrough, `docs/production-model.md` as the operating-model reference, and `docs/adoption-api-contract.md` as the only public API contract. Entry docs should link to those canonicals instead of restating overlapping guidance.
- S02 / Runtime truth for production framing: In M001, tenant, policy, API key, and operator admin surfaces are the real enforced entities; `app` and `workload` are documentation guidance only and should not be described as first-class runtime objects.
- S02 / Verification environments: Slice-level verification for adoption docs depends on both Python test tooling (`pytest`) and console test tooling (`console/node_modules/.bin/vitest`) being provisioned in the active worktree. Missing local toolchains produce environment gaps, not product evidence.
- S03 / Migration proof pattern: The canonical reference migration proof should correlate a public `POST /v1/chat/completions` response to operator evidence by pairing `X-Request-ID` and `X-Nebula-*` response headers with `GET /v1/admin/usage/ledger?request_id=...`.
- S03 / Tenant header rule in practice: `X-Nebula-Tenant-ID` is not part of the default migration diff; require it only for API keys intentionally authorized for multiple tenants (`tenant_id=None` with multiple `allowed_tenant_ids`), otherwise treat tenant inference from the key as the happy path.
- S03 / Runtime-aligned governance assertions: In the current runtime, admin policy options report `gpt-4o-mini` as the default premium model and the explicit premium-request ledger scenario in governance coverage records `policy_denied`/`denied`; keep slice verification assertions aligned to those observed defaults unless the runtime contract changes again.
