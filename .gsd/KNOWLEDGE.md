# Knowledge

- S01 / API contract: Treat `docs/adoption-api-contract.md` as the only canonical public compatibility boundary. Update README and architecture docs by linking back to it rather than restating request/response details in multiple places.
- S01 / Validation behavior: `ChatCompletionRequest` now enforces the "at least one user message" invariant at schema-validation time, so invalid requests return FastAPI's native 422 `detail` payload and do not emit route-level `X-Nebula-*` headers.
- S01 / Test alignment: The admin Playground is an admin-only, non-streaming operator surface that can route differently from the public adoption path for the same requested model. Do not assume Playground responses should mirror public-route provider selection.
- S01 / Header assertions: For routed denials and fallback-blocked paths, assert that `X-Nebula-Policy-Outcome` is present and meaningful rather than hard-coding a specific string unless the runtime contract explicitly guarantees that exact value.
- S02 / Adoption docs composition: Keep `docs/quickstart.md` as the single happy-path self-hosted walkthrough, `docs/production-model.md` as the operating-model reference, and `docs/adoption-api-contract.md` as the only public API contract. Entry docs should link to those canonicals instead of restating overlapping guidance.
- S02 / Runtime truth for production framing: In M001, tenant, policy, API key, and operator admin surfaces are the real enforced entities; `app` and `workload` are documentation guidance only and should not be described as first-class runtime objects.
- S02 / Verification environments: Slice-level verification for adoption docs depends on both Python test tooling (`pytest`) and console test tooling (`console/node_modules/.bin/vitest`) being provisioned in the active worktree. Missing local toolchains produce environment gaps, not product evidence.
- S03 / Migration proof pattern: The canonical reference migration proof should correlate a public `POST /v1/chat/completions` response to operator evidence by pairing `X-Request-ID` and `X-Nebula-*` response headers with `GET /v1/admin/usage/ledger?request_id=...`.
- S03 / Tenant header rule in practice: `X-Nebula-Tenant-ID` is not part of the default migration diff; require it only for API keys intentionally authorized for multiple tenants (`tenant_id=None` with multiple `allowed_tenant_ids`), otherwise treat tenant inference from the key as the happy path.
- S03 / Runtime-aligned governance assertions: In the current runtime, admin policy options report `gpt-4o-mini` as the default premium model and the explicit premium-request ledger scenario in governance coverage records `policy_denied`/`denied`; keep slice verification assertions aligned to those observed defaults unless the runtime contract changes again.
- S04 / Day-1 proof ordering: Preserve the proof flow as public `POST /v1/chat/completions` request first, then `X-Nebula-*` headers plus `X-Request-ID`, then usage-ledger correlation, with Playground and Observability used only as operator corroboration surfaces.
- S04 / Operator-surface framing: For adoption-trust work, Playground should show immediate request metadata while Observability should read as the persisted explanation surface for route/provider/fallback/policy plus dependency-health context; do not collapse those roles into a single generic admin view.
- S05 / Final integrated proof wording: Keep the operator proof sequence explicit in both UI copy and browser assertions — Playground is immediate corroboration for the same request ID, while Observability is the persisted explanation and dependency-health view reached only after the public X-Request-ID / X-Nebula-* correlation story is established.
- S05 / Verification-gap handling: Treat missing local runners or missing targeted test files as environment gaps, but treat unrelated compile failures that block a broader runner (for example Playwright webServer startup failing in an untouched console area) as separate worktree blockers rather than evidence that the proof surfaces changed in this slice are broken.
- M001 / Close-out integrity rule: Requirement transitions can remain historically validated when backed by slice evidence, but milestone close-out must still fail if the final worktree snapshot deletes canonical docs or executable proof artifacts that the assembled adoption story depends on.
- M002 / Console truth-surface guardrail: When console copy explains operator scope or evidence, derive the wording from existing runtime/admin contracts instead of inventing new fields or entities — API key scope should mirror `tenant_id` + `allowed_tenant_ids`, Playground should stay admin-only/non-streaming corroboration, and Observability should stay the persisted request-evidence plus dependency-health context surface, with focused vitest assertions locking that boundary.
- M002 / Tenant app-workload guidance pattern: On tenant-facing surfaces, explain app/workload mapping only as conventions carried through tenant names, API key names, and optional notes; pair those positive assertions with explicit negative checks against `workspace`, `app`, or `workload` pseudo-entity drift in focused Vitest coverage.
- M002 / Integrated proof ordering guardrail: Keep the final production-structuring walkthrough composition-first and preserve the order `production-model` structuring → public `POST /v1/chat/completions` → `X-Request-ID` and `X-Nebula-*` headers → usage-ledger correlation → Playground corroboration → Observability persisted explanation. If a future change starts with Playground or treats Observability as a replacement for ledger/public evidence, that is a regression in the adoption story rather than a harmless copy tweak.
- M002 / Milestone close-out verification pattern: For documentation-and-copy-heavy milestones, do not rely only on prior slice summaries. Re-run the focused contract tests in the assembled worktree, confirm integrated proof artifacts are still present, and verify the diff against `main` includes non-`.gsd` changes before marking the milestone complete.
- M002 / Requirement discipline for guidance-only work: A milestone can materially strengthen a deferred requirement's evidence without changing its status. If the work intentionally improves conceptual guidance without introducing the first-class runtime/admin capability named by the requirement, keep the requirement deferred and record the stronger proof explicitly.
- M003 / Embeddings route wiring pattern: Public embeddings should stay on a dedicated `/v1/embeddings` route that calls the embeddings service directly and uses request-scoped accessors or `request.app.state.container` for dependencies/ledger recording, rather than reusing chat orchestration or `Depends(get_container)`, which risks FastAPI treating helper parameters as request inputs.
- M003 / Embeddings evidence pattern: Reuse the existing usage ledger for embeddings adoption proof by correlating `X-Request-ID` to metadata-only rows (`final_route_target="embeddings"`, provider/status/reason/policy fields) and keep raw input text plus returned vectors out of persistence.
- M003 / Embeddings docs composition: Keep `docs/embeddings-adoption-contract.md` as the only detailed public embeddings boundary. README and architecture docs should link to it for discoverability and endpoint inventory, but should not restate request/response, header, or failure-semantics details inline.
- M003 / Embeddings migration proof pattern: Treat `tests/test_embeddings_reference_migration.py` as the source of truth for the realistic caller swap; the proof is only complete when the same public `/v1/embeddings` response yields exact `X-Request-ID`/`X-Nebula-*` assertions and a matching metadata-only usage-ledger row from `GET /v1/admin/usage/ledger?request_id=...`.
- M003 / Embeddings observability pattern: Keep embeddings evidence discoverable by extending the shared usage-ledger filter/table/detail surfaces (`routeTarget -> route_target`, visible `request_id`, persisted route/model/outcome fields) instead of adding route-specific UI flows or a new admin API; Observability should remain corroboration of the public `X-Request-ID` plus ledger proof, not a replacement for it.
- M003 / Final assembly composition rule: The integrated embeddings walkthrough should stay composition-first and pointer-only. Use it to join the proof path, but keep detailed contract semantics in `docs/embeddings-adoption-contract.md`, realistic caller diff details in `docs/embeddings-reference-migration.md`, and treat README/architecture discoverability links as pointers rather than alternate contract surfaces.
- M003 / Milestone close-out verification rule: For milestone close-out, verify non-`.gsd` code/doc changes against the integration branch named in `.gsd/milestones/<id>/<id>-META.json`, not just against `main`. A diff against the wrong base can falsely suggest milestone artifacts were deleted even when the current assembled worktree is correct.
- M003 / Embeddings adoption proof rule: The milestone is only truly assembled when all four layers remain present together in the final worktree — public runtime route, canonical contract doc, executable migration proof, and shared Observability corroboration surfaces. If any one of those disappears, treat it as milestone-proof regression even if slice summaries still look complete.
- M005 / Route-signals ledger mapping: When adding fields to `UsageLedgerRecord`, update both `GovernanceStore.record_usage()` and `_usage_from_model()` together; otherwise admin ledger responses can silently drop persisted fields even though the database write path and migration are correct.
- M005 / Policy simulation replay contract: `PolicySimulationService` intentionally replays newest-fetched ledger rows oldest-first so replay windows, aggregate counts, and bounded changed-request samples are deterministic and chronological. In tests, assert the changed sample against that oldest-first contract rather than assuming database fetch order.
- M005 / Simulation change detection: A replay can count as "changed" even when the route target stays the same if the simulated policy outcome or projected premium cost changes. Focus assertions on route/status/policy/cost deltas instead of treating changed-request samples as route-target-only evidence.
- M005 / Hard-budget contract: Represent hard cumulative guardrails with explicit `hard_budget_limit_usd` and `hard_budget_enforcement` policy fields and keep `soft_budget_usd` advisory-only in both backend metadata and console copy; rewriting `soft_budget_usd` into enforcement semantics causes contract drift across runtime, simulation, and UI surfaces.
- M005 / Hard-budget replay semantics: Shared hard-budget evaluation uses `tenant_spend_total` plus `before_timestamp`, so replayed requests intentionally exclude their own ledger row and derive thresholds from real persisted premium cost rather than fixture assumptions about token settings.
- M005 / Recommendation contract discipline: Keep recommendation feedback bounded and typed through `RecommendationBundle` / `RecommendationCard` / `CacheControlSummary`, sourced only from usage-ledger evidence plus coarse semantic-cache runtime health. Do not widen this seam into free-form analytics, raw prompt inspection, or cache-entry introspection.
- M005 / Cache tuning surface rule: Semantic cache similarity threshold and max entry age are runtime-enforced tenant policy controls, but editing must stay in the existing policy save/preview flow. Observability may inspect and explain cache effectiveness and runtime state, but it should remain read-only and should not become a separate cache-management surface.
- M005 / Migration hygiene for policy fields: When TenantPolicy gains persisted fields, add the matching Alembic revision in the same slice. ORM-only changes can leave governance API tests failing at startup before new endpoint behavior is exercised.
- M005 / Final v4 proof composition rule: Keep `docs/v4-integrated-proof.md` pointer-only and composition-first. Use it to join route vocabulary, policy replay, hard guardrails, bounded recommendations, cache controls, Observability corroboration, and benchmark evidence in one strict review order, but keep detailed contracts in their existing canonical docs/code surfaces and verify discoverability with explicit file/link checks plus focused runtime/UI tests rather than long prose snapshots.
- M005 / Milestone close-out verification rule: Verify the assembled v4 story end to end before marking the milestone complete — confirm real non-`.gsd` code/doc diff, rerun the integrated backend/admin/console checks, and treat pointer-only proof discoverability (`docs/v4-integrated-proof.md` plus README/architecture links) as part of the shipped contract, not optional documentation garnish.
- M005 / Requirement discipline at close-out: It is valid to finish the milestone with R039 still active while validating R044. Improved routing evidence and route-score visibility materially strengthen the control plane, but requirement status should only flip to validated when the assembled behavior proves the stronger claim — in this case the narrow, operator-driven v4 scope constraint was fully proven while outcome-aware routing remains future work.
- M006 / Calibrated routing override boundary: Explicit model overrides and policy-forced routing (`explicit_*`, `policy_*_only`) intentionally keep `route_signals` empty and `route_score` at `0.0`; only `token_complexity` decisions emit the calibrated/degraded signal payload and additive score breakdown.
- M006 / Policy denial header parity: When policy evaluation denies a routed request, emit the same `X-Nebula-Route-Score` and conditional `X-Nebula-Route-Mode` semantics as the normal/error metadata path. Otherwise denial responses drift from persisted route evidence and focused header tests miss real contract regressions.
- M006 / Response-header test discipline: In calibrated routing tests, derive expected `token_count` / `token_score` from the actual prompt fixture in the test rather than carrying forward stale numeric expectations. Small fixture-text changes legitimately move additive score components even when the contract is stable.
- M006 / Calibration summary staleness rule: Tenant calibration evidence staleness is based on the newest eligible ledger row, not only the newest sufficient/calibrated row. Fresh degraded eligible evidence keeps the summary from becoming `stale`, while explicit overrides and `calibrated_routing_disabled` rows stay excluded from freshness math.
